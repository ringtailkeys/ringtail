---
title: MCP tools
description: The exact tool surface a coding agent drives over the daemon's MCP connection. Every tool returns names + status, never a value.
---

These are the tools the daemon exposes over MCP (Streamable HTTP, `127.0.0.1/mcp`, session
token required). The whole surface is the generative-UI vocabulary: the agent supplies
structured content; the daemon owns the pixels and the creds. **No tool returns a secret
value** ‚Äî that is enforced by `check:no-leak`.

## Plan & status

- **`plan()` ‚Üí `Status[]`** ‚Äî scan the manifest + connected state ‚Üí the grid (providers √ó
  `local/dev/staging/prod`). Emits key **names** and status, never values. Also carries
  `pendingUserMessages` ‚Äî see [Chat](#chat-the-direction-channel).
- **`updateStatus(provider, env, status)`** ‚Äî flip one grid cell to a credential status.

## Author & render the UI

- **`authorWizard(provider)` ‚Üí `Wizard`** ‚Äî derive the curated recipe's on-brand 1-2-3 setup
  wizard (`open-url` ‚Üí `paste` ‚Üí `provision`) and push it to the cockpit. Returns step names +
  kinds.
- **`renderWizard(wizard)`** ‚Äî push an agent-authored `Wizard` (the long-tail fallback) to the
  cockpit. Schema-validated; malformed is rejected.
- **`mapActions(actions)` ‚Üí `Action[]`** ‚Äî map the repo-specific + cross-tool next steps into
  the actions panel. Each `Action` is schema-validated. Returns names + prerequisites + danger
  + executor.
- **`renderActions(actions)`** ‚Äî re-render the living action panel as the user steers it.

## Complete & execute

- **`submitStep(stepId, value?)`** ‚Äî the **one** inbound value path. For a `paste` step the
  value flows user ‚Üí daemon (validated + stored) and is **never echoed back**. Returns status +
  var name only. When the next step is a safe `auto` step, the daemon **auto-advances** it
  (mint ‚Üí validate-after-mint ‚Üí provision ‚Üí sync) with no agent round-trip ‚Äî the result carries
  the value-free `autoAdvanced` outcome.
- **`executeStep(stepId)`** ‚Äî the daemon runs the provisioning loop (mint ‚Üí validate-after-mint
  ‚Üí provision ‚Üí sync) with the stored creds. Returns status + key names.
- **`executeAction(id, confirmed?)`** ‚Äî run a mapped action through the gates (prereqs ‚Üí
  hard-confirm ‚Üí dispatch). A `destructive` action is refused unless `confirmed: true`. Returns
  status + key names.
- **`mintKey(action, env?)`** ‚Äî the **generic dynamic executor**. The agent authors
  an HTTP action inline ‚Äî `{ providerAccount, method, url, headers (with a {{ROOT}} placeholder),
  body?, extract? }` ‚Äî and the daemon resolves the root key from the global vault, **enforces the
  domain allowlist** (a non-allowlisted host is rejected before any HTTP; redirects off the
  allowlisted host are refused, not followed), runs the call, and ‚Äî if `extract` names a minted
  secret ‚Äî files it in the sink under `ringtail/<repo>/<env>/<provider>` naming. One path covers
  **mint**, read-only **permission-check** (no `extract`), and cross-provider **wire**. A
  consequential action (e.g. a `DELETE` that spends the root key destructively) comes back
  `needs-confirm` ‚Äî the agent gets **no** `confirmed` argument, so it can never self-approve; a
  human confirms out-of-band. Returns `{ providerAccount, varName?, status, reason? }` ‚Äî **never a
  value** (root or minted; any reflected secret is scrubbed from `reason`). The root key itself is
  pasted by the user via `POST /api/root`, never through the agent.
- **`provisionProject(mints, vars?, env?)`** ‚Äî **the batch orchestrator** (the repetition-killer).
  *A new project provisions itself.* The agent authors **one batch** of mint/wire actions ‚Äî one per
  var that self-provisions from a connected root (each a normal [`mintKey`](#complete--execute)
  template: a `{{ROOT}}` mint with `extract`, or a wire action like GoDaddy set-nameservers with no
  `extract`) ‚Äî plus the project's full `vars` list (defaults to the active grid). The daemon parks
  the **whole batch under a single unforgeable nonce** ("provision these N keys for &lt;project&gt;?")
  and classifies the vars **not** in the batch with the value-free planner into four buckets:
  **`mint-from-root`** (a connected root + a recipe exist), **`needs-root`** (a recipe but no root ‚Äî
  connect it once, then it self-provisions), **`guided-paste`** (no recipe ‚Äî paste it by hand), and
  **`skip`** (a non-secret provisioned *resource* like `DATABASE_URL` ‚Äî needs a resource, not a key;
  flagged, never faked). It returns `needs-confirm` + the value-free plan + the parked list ‚Äî **never
  a value**. On the human's **one** approval (`POST /api/action` with the nonce) every action runs
  through the **same** mint gate; each var reports its own value-free result (`minted` / `ok` /
  `no-root` / `failed`). Partial success is fine. The agent can never self-approve (it never receives
  the nonce), and a structurally-doomed action rejects the whole batch up front.
- **`rotateKey(rotate, env?)`** ‚Äî **rotate a credential** (mint a fresh scoped key ‚Üí switch the
  sink to it ‚Üí revoke the old key) as **one human-approved, atomic operation with safe rollback**.
  The agent authors `{ varName, mint, revoke }`: `mint` is a normal scoped/guided
  [`mintKey`](#complete--execute) template (its `extract.varName` must equal `varName`; its
  `extract.idPath` names the new key's id), and `revoke` is the old-key delete ‚Äî its url/body may
  carry `{{OLD_KEY_ID}}` (filled daemon-side from the stored old key id) and its headers carry
  `{{ROOT}}`. A rotation is inherently consequential (it mints **and** revokes with the root key),
  so it comes back `needs-confirm` and is parked under an unforgeable nonce ‚Äî the agent can never
  self-approve. On the human's one approval the daemon runs the whole rotation **locally**:
  **mint-new ‚Üí reconfigure the sink ‚Üí (optional validate) ‚Üí revoke-old**, with safe rollback ‚Äî
  if mint or the sink write fails it **aborts and restores the old working key** (never revoking),
  and if the revoke fails after the sink switched it returns **`partial`** (new key live, "revoke
  the old one manually"). Returns names + status only ‚Äî **never a value** (old or minted).
- **`mintViaBrowser(provider, varName, env?)`** ‚Äî **browser mint** (Envoyage), the *no-mint-API*
  path. For a **dashboard-only** provider (no management API to mint a token), the daemon drives the
  provider's **web console with a real browser** to create + read back the key. It drives **headless
  until it can't**, then STOPS at any genuine human wall ‚Äî **login / CAPTCHA / OTP**: the human
  solves it in the **live view**, and the agent is **structurally blind** to the password (Envoyage
  blanks the screenshot on a password page, so the model gets text only). A browser mint creates a
  real credential, so it is consequential: it comes back `needs-confirm` and is **parked under an
  unforgeable nonce** ‚Äî the agent can never self-approve. On the human's approval (`POST /api/action`
  with the nonce) the daemon drives the mint **locally** through the **detect ‚Üí pause ‚Üí wait-for-human
  ‚Üí resume** handoff state machine, captures the value daemon-side, and closes the loop through the
  **same validate + sink** as an API mint. Returns the var **name** + status only ‚Äî **never a value**.
  OFF unless `RINGTAIL_BROWSER_MODE` is `local` (local Chromium, OSS) or `cloud` (a Cloudflare browser
  via cdp-url, paid). The planner surfaces the matching var as the **`mint-via-browser`** bucket.

## Connect providers

- **`listConnectors()`** ‚Äî the agent-guided onboarding surface. Lists the OAuth providers the user
  can **connect** (id, `connected?`, `needsClientCreds?`, `scopes`, and the signup / api-keys URLs
  to guide the user to), plus the already-connected providers (names + scopes + expiry). The
  dashboard drives the actual connect (`POST /api/connect/start` opens the authorize URL); this
  just lets the agent say "sign up / connect here". **Never returns a token.**

## Chat (the direction channel)

- **`sendChat(message, choices?)`** ‚Äî agent ‚Üí user in the dashboard chat, optionally with
  tappable choice pills. Text/intent only, never a value.
- **User ‚Üí agent is event-driven (no poll tool).** A message the user types (`POST /api/chat`)
  is queued and delivered as **`pendingUserMessages: string[]`** on the return of the agent's
  next `plan` / `executeStep` / `updateStatus` / `authorWizard` call ‚Äî intent text only, never a
  value. The agent re-renders actions/wizards to match on its next natural tool call.

<Callout type="warn">
Notice what's absent: there is no `getSecret`, no `readValue`, no tool that hands the agent a
credential. The agent's entire world is names, statuses, and the UI it authors. See
[The guarantee](/docs/concepts/the-guarantee).
</Callout>
